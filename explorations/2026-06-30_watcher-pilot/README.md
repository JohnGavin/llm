# Watcher Pilot — 2026-06-30

**Package:** r-lib/watcher v0.2.0 (native-OS filesystem event monitor via libfswatch)
**Issue:** JohnGavin/llm#696
**Status:** PASS — nix build succeeded, callback fired

## Purpose

Test whether `watcher` (r-lib/watcher, a compiled R binding for libfswatch) can
be installed from GitHub source via rix `git_pkgs` under our 2026-02-02 nixpkgs
pin, and whether its event-driven filesystem callback mechanism works in practice.
The compiled nature of the package (libfswatch bundled in `src/fswatch/`) was the
primary build risk.

## Outcome

### Nix Build Status: SUCCESS

The package compiled cleanly under clang 21.1.8 with MacOSX14.4.sdk. Key facts:

- libfswatch is **vendored** in the package (`src/fswatch/libfswatch/`). No external
  `libfswatch` system package is needed in nixpkgs (none exists there anyway).
- On macOS, the configure script appends `-framework CoreServices` automatically
  via `case $(uname -s) in Darwin)`. The CoreServices framework resolved via the
  macOS SDK already present in the nix clang wrapper (`MacOSX14.4.sdk`).
- Rix's `buildRPackage` expression does not need extra `buildInputs` for this
  package on macOS — the default stdenv supplies the SDK.
- The rix-generated `default.nix` used `fetchgit` for commit
  `ab79639335c519a9cc7eeca272ea744ade462329` (v0.2.0 tag, released 2026-06-22).

### Callback Demo: SUCCESS

The `watcher_demo.R` spike verified the full event cycle:

1. `watcher(path, callback, latency = 0.2)` created successfully
2. `$start()` returned TRUE; `$is_running()` TRUE
3. Writing `trigger.txt` inside the watched directory triggered the FSEvents monitor
4. The callback fired within ~200ms (the debounce latency), confirmed by
   `later::run_now()` pumping the event loop
5. `$stop()` cleanly terminated the monitor

One behavioural note: on macOS, FSEvents reports the **containing directory** as
the changed path (e.g. `watcher_test_84628/`), not the specific file
(`trigger.txt`). This is expected FSEvents semantics — the callback receives a
character vector of directories where changes occurred, not file paths. Callers
needing file-level paths must scan the directory themselves.

## Watcher vs `/loop <interval>` — Verdict

| Dimension | `/loop <interval>` (polling) | watcher (event-driven) |
|---|---|---|
| Latency | Interval-bound (e.g. 30s minimum practical for R CMD check) | Near-instant (FSEvents, ~200ms debounce) |
| CPU when idle | Wakes every interval | Zero overhead between events |
| Missed events | If event happens just after a check, waits full interval | None (all events captured) |
| Complexity | One line: `/loop 30s /check` | Requires R session, later event loop pump |
| Use in Claude Code | Native slash command, no extra setup | Needs a persistent R process |
| Suitability for "rerun on save" | Acceptable for slow operations (R CMD check) | Better for fast feedback (document/test) |

**Adopt verdict: CONDITIONAL.** Watcher is technically sound and builds cleanly in
nix. The event-driven callback is genuinely faster than polling — a 0.2s debounce
vs. a 30s minimum `/loop` interval is meaningful for fast feedback loops (lint,
`devtools::document()`). However:

- It requires a persistent R process to pump the `later` event loop.
- For the specific Claude Code use case (`/loop /check`), the bottleneck is R CMD
  check runtime (minutes), not the polling interval. Switching to event-driven here
  gives no practical benefit.
- The FSEvents directory-granularity (not file-granularity) means callers need to
  filter events themselves.

**Recommended use case:** watcher is the right tool for Shiny dev-reload (which is
its original purpose) and for sub-second file-change detection inside long-running
R processes. For `/loop`-style CI workflows, polling remains simpler. If we later
build a "run tests on file save" feature inside R (not via Claude Code slash
commands), watcher is the right engine.

**Not yet a production adoption** — the exploration shows the package works but the
specific `/loop` replacement use case does not justify the added complexity.
Graduate to `R/` if a concrete "run-on-save" feature is planned.
